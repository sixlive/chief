## Pre-Commit Audit (MANDATORY — run all five before committing)

A separate reviewer agent will inspect your diff after you commit. The reviewer rejects work for the gap patterns below. Catch them yourself first.

For EACH audit item, write a short note in your progress report showing the check you ran and what it found. "I looked" is not an audit; show the grep, name the test, point at the file.

1. **Deprecated path audit.** If you added a new variant of an existing function, method, query, endpoint, or type (e.g., a thread-aware version of a user-wide query), grep for callers of the OLD form. Either update each caller, or document why each one is intentionally untouched. Adding a new path beside an old one and leaving both callable is a rejection.

2. **Adversarial test audit.** If this story touches any endpoint, query, or resource keyed by `user_id` / `tenant_id` / owner, name (a) the ownership check that prevents cross-tenant access and (b) the test that proves a different user gets denied. If either is missing, fix it before committing.

3. **Observability audit.** For EACH acceptance criterion, name the observable test that proves it — an end-to-end test, a UI assertion, an HTTP request/response check, a browser test. If the only proof is "a row exists in the database," the criterion is satisfied at the wrong layer; rewrite the test to assert what the user (or an external observer) actually sees.

4. **Invariant audit.** Re-read the Global Invariants block above. For EACH invariant, state how this story upholds it, OR explicitly state that the invariant is untouched by this story (with a one-line justification). Silent invariant violations are the most expensive class of bug.

5. **Half-wiring audit.** Open the PRD at §1 (Problem Statement) and read the defect class the PRD exists to eliminate. For EACH anti-pattern named there, run the tree-wide grep — **not just against your diff** — and report the count. An anti-pattern surviving in an unchanged file is still a half-wiring. If an invariant includes a negative-fingerprint grep, run it verbatim. Zero matches is the bar; explain any remaining matches (and why they are not in scope for this story, if truly so). Guardrail / no-regression tests added or touched by this story must also pass an oracle check: if the function under test returned a constant, would the test still pass? If yes, the oracle is tautological — rewrite it to derive the expected value from the inputs.
