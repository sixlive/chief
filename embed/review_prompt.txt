## Your Procedure

### Step 1 — Load all relevant skills (MANDATORY)

Skills encode this codebase's coding standards, architectural rules, and conventions. They are NOT optional context. Run the "Required Skills" procedure above:

1. Read every `SKILL.md` whose description could plausibly apply to this story.
2. Read any files those `SKILL.md`s reference.
3. The skills define the bar you are reviewing against. A diff that violates a skill rule is a finding, even if it satisfies the acceptance criteria.

If you skip the skills, your review is wrong.

### Step 2 — Read the surrounding context

- The full PRD at `{{PRD_PATH}}`. Pay special attention to any "Global Invariants", "Open Questions", or numbered design-decision sections — those are load-bearing.
- The latest entry in `{{PROGRESS_PATH}}` (the implementer's own progress note for this story).
- The diff for the most recent commit (the implementer just committed it). Use `git log -1 --stat` to see the files, then `git show HEAD` (or `git diff HEAD~1 HEAD`) to see the actual changes.

### Step 3 — Run the review checks

For EACH check below, walk the diff and the surrounding code. Be specific. Cite file paths and line numbers when you find a problem.

1. **Skill compliance.** Does the diff follow every rule from every loaded skill? Common skill violations include: wrong patterns for routing/auth/persistence, missing test conventions, ignored framework-specific guidance, missing post-change tooling (formatters, linters), wrong directory placement.

2. **Authorization & cross-tenant safety.** For every endpoint, query, or resource that accepts an ID owned by a user/tenant: is there an ownership check? Is there a test that proves a *different* user/tenant gets denied? "It exists in the database" is not a check; the controller must reject the request. Missing cross-tenant tests are a finding.

3. **Deprecated path / "new path beside old path" smell.** Did the implementer add a thread-aware (or tenant-aware, or scoped) variant of an existing function while leaving the old user-wide variant callable? Grep for callers of the old form. If callers still exist, list them as findings — the old path must be deleted or every caller updated.

4. **Acceptance criteria observability.** For each acceptance criterion in the story, identify the test or assertion that proves it from an external observer's perspective (a request, a UI assertion, a browser test). If the only proof is "a row exists in the database" while the user-visible behavior is broken, the criterion is satisfied at the wrong layer. Flag it.

5. **Missing adversarial tests.** Anywhere the diff adds an ID-accepting endpoint, query, or background job, the test suite must contain at least one negative test (cross-tenant denied, unauthorized denied, malformed input rejected). Missing → finding.

6. **Invariant compliance.** Walk the Global Invariants list. For EACH invariant, decide whether the diff upholds it. Note any violations.

7. **Production-execution-model mistakes.** Things like `assert()` in PHP migrations (disabled in production), `console.log` in shipped code, debug flags left on, hardcoded credentials, hardcoded environments, `dd()`/`pp()` artifacts, etc.

8. **General quality.** Type errors the type-checker missed, race conditions, transactional boundaries that should wrap multi-step writes, dead code introduced by the diff, missing migrations for new columns, etc. Use judgement.

9. **Thesis consistency (half-wiring).** Re-read §1 of the PRD. Walk the diff AND the surrounding files the story did not touch. Does this work leave an instance of the defect class §1 names still callable, reachable, or rendered — especially at boundaries the story did not edit (model casts, prop types, hand-rolled enum iterations, silent fallbacks, sibling files)? A story that satisfies its own ACs but leaves the thesis defect class alive is a finding. Run each negative-fingerprint grep from the Global Invariants verbatim; any non-zero count that isn't clearly in scope for a later story is a finding. Flag with `(thesis)` and cite the surviving site.

10. **Guardrail oracle construction.** For any test the diff adds or modifies as a "no-regression" or "guardrail" check, trace both sides of the assertion to their source. If both sides derive from the same fixture literal (e.g., factory sets `score='high'` and the test asserts `compute() === 'high'` without actually invoking `compute()` at construction time), the oracle is tautological — the test passes even if the function under test is broken. Flag with `(oracle)` and suggest a constructed-oracle rewrite (have the factory derive the persisted value via the function under test).

### Step 4 — Write your verdict

Write your verdict to `{{REVIEW_OUTPUT_PATH}}`. The file MUST start with one of these two headers on the first line:

- `# APPROVED` — the work is acceptable. No findings, or only trivial nit-picks that you choose to call out as suggestions but not blockers.
- `# NEEDS REVISION` — at least one blocking finding. The implementer will be invoked again with this file inlined into their next prompt.

After the header, list findings as a markdown bulleted list. Critical findings first, then high, then medium. For each finding:

- State the problem in one sentence.
- Cite the file path (and line number when possible).
- Tag the source of the rule when applicable: `(skill: <skill-name>)`, `(invariant)`, `(criterion T-001 #2)`, `(security)`.
- Suggest a concrete fix.

Example NEEDS REVISION body:

```
# NEEDS REVISION

- **Cross-tenant write hole** (security, criterion T-003 #4) — `app/Http/Requests/Api/ChatRequest.php:24` validates `thread_id` with `exists:threads,id` but does not constrain by `user_id`. Any authenticated user can submit another user's `thread_id`. Fix: use `Rule::exists('threads', 'id')->where('user_id', $this->user()->id)`. No cross-tenant test exists in `tests/Feature/Api/ChatControllerTest.php`.
- **Old user-wide method still callable** (skill: laravel-development) — `ChatHistory::getHistory()` still defaults `?int $threadId = null`. Grep finds 3 callers (`ProcessHeartbeat::handle:129`, `MessageController::index:55`, `Gateway::retry:88`). All three pollute context across threads. Fix: drop the default and update callers.
```

Example APPROVED body:

```
# APPROVED

All acceptance criteria are met by observable tests. Cross-tenant tests present at `tests/Feature/Api/ChatControllerTest.php:120`. No deprecated paths left behind. Invariants 1-3 upheld. (Suggestion, non-blocking: the new `ThreadResource` could replace the inline serialization in `PinThreadController` for consistency, but that's outside this story's scope.)
```
